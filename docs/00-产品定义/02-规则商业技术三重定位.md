# 第 2 章 · 规则、商业、技术三重定位

> 本章为系统的三重底座:**规则(合规) · 商业(增长) · 技术(架构)**。三者必须同时满足,任一缺失都会让系统从"赋能工具"退化为"高级网推所"。

---

## 2.1 规则维度:把律师职业边界写进系统

### 系统必须内置的十条底线

| # | 规则 | 强度 |
|---|------|------|
| 1 | 律师事务所 / 律师是法律服务交付主体 | **强** |
| 2 | 运营人员、客服人员、法律服务支持企业 **不得作法律判断** | **强** |
| 3 | **不得承诺办案结果** | **强** |
| 4 | **不得宣示胜诉率、赔偿额、回款率** | **强** |
| 5 | 不得明示或暗示与司法机关、行政机关、仲裁机构有特殊关系 | **强** |
| 6 | 不得低价诱导后续不透明加价 | 中 |
| 7 | 不得以案件结果、律师费、回款额向第三方分成 | **强** |
| 8 | **不得将客户隐私、商业秘密和案件敏感信息外泄** | **强** |
| 9 | **AI 不得替代律师作出最终法律意见** | **强** |
| 10 | **客户侧数字人不得自动回答实质法律判断问题** | **强** |

### 法规依据(摘要)

- 律师业务推广规则中对虚假、误导性或者夸大性宣传、承诺办案结果、宣示胜诉率、赔偿额、标的额等行为均设有禁止性要求;
- 律师法亦要求律师保守执业中知悉的国家秘密、商业秘密和当事人隐私;
- 同时,客户信息、案件材料、语音记录、满意度反馈等均涉及个人信息处理,系统应按照个人信息保护规则采取必要措施保障个人信息安全。

## 2.2 商业维度:从"流量转化"转向"合规增长"

附件文章提出,"**网推所**"通常表现为 **流量前置、销售前置、收费前置和责任外部化**,其危险在于用销售链条替代专业判断和责任承接。

### LM Unity · Counsel 的商业逻辑必须相反

| 网推所模式 ❌ | 本系统模式 ✅ |
|--------------|--------------|
| 流量前置 | **可信身份前置** |
| 销售前置 | **结构化分诊前置** |
| 收费前置 | **透明报价前置** |
| 责任外部化 | **律所 / 律师责任闭环** |

### 商业增长标准流水线(15 步)

```
内容合规
  → 线索进入
  → 律时初步接待
  → 结构化分诊
  → 律师确认
  → 服务产品推荐
  → 报价卡
  → 风险揭示
  → 服务边界确认
  → 委托签约
  → 案件办理
  → 客户成功
  → 结案复盘
  → 知识沉淀
  → 复购 / 转介绍
```

## 2.3 技术维度:从工具集合升级为法律服务操作系统

### 系统 **不是** 以下工具的简单拼接

- ❌ CRM
- ❌ OA
- ❌ 语音助手
- ❌ 文档库
- ❌ 聊天机器人
- ❌ 案件管理软件
- ❌ 营销系统

### 而应是一个 **统一的"法律服务操作系统"**

| # | 子系统 | 核心能力 |
|---|--------|----------|
| 1 | **合规增长中台** | 内容审核、线索分诊、营销合规、风险监控 |
| 2 | **律时数字人** | 对客咨询、对内助理、合规话术 |
| 3 | **客户生命周期系统** | 线索 → 签约 → 办案 → 复购全旅程 |
| 4 | **法律服务产品库** | 服务产品化、套餐定价、报价卡生成 |
| 5 | **案件办理系统** | 立案、任务、协同、文档、证据链 |
| 6 | **时间计费系统** | 工时记录、账单生成、收费对账 |
| 7 | **质量控制系统** | SLA 监控、案件质量评分、客户回访 |
| 8 | **知识沉淀系统** | 案例库、模板库、判例库、专家图谱 |
| 9 | **数据安全与审计系统** | 分级分类、加密、审计日志、合规留痕 |
