# 第 5 章 · 权限模型

> 本章定义 LM Unity · Counsel + 律时 的 **7 维权限模型** + **6 级数据密级** + **运营 / 客户两侧权限边界**。
>
> 权限模型是 [2.1 合规底线](./02-规则商业技术三重定位.md)、[3.2 数据隔离](./03-产品总架构.md)、[4.x 角色体系](./04-用户角色体系.md) 的具体落地机制,也是后续权限中间件、API 网关、审批流引擎的 **工程输入**。

---

## 5.0 权限模型全景

```
权限  =  RBAC  +  ABAC  +  TenantScope  +  MatterScope  +  DataLevel  +  StageGate  +  ApprovalFlow
        (角色)    (属性)    (租户隔离)      (案件级)       (数据密级)     (阶段门禁)     (审批流)
```

7 个维度缺一不可,共同决定 **"谁,在什么数据范围内,能做什么动作,且需经过什么审批、处于什么阶段"**。

---

## 5.1 权限设计原则:7 维模型

| # | 组件 | 含义 | 解决的问题 |
|---|------|------|------------|
| 1 | **RBAC** | 角色权限 | "我是谁"——基于 [第 4 章 33 角色](./04-用户角色体系.md) |
| 2 | **ABAC** | 基于属性的权限 | "我在什么条件下"——时间、地点、设备、风险评级等属性 |
| 3 | **TenantScope** | 租户隔离 | "我能看哪个租户"——[3.2 节 4 级数据隔离](./03-产品总架构.md) |
| 4 | **MatterScope** | 案件级权限 | "我能看哪些案件"——具体到 `matter_id` |
| 5 | **DataLevel** | 数据密级 | "我能看什么密级数据"——[5.2 节 6 级分类](#52-数据密级6-级) |
| 6 | **StageGate** | 流程阶段门禁 | "当前阶段是否允许"——案件 / 业务流程阶段校验 |
| 7 | **ApprovalFlow** | 审批流 | "需要谁审批"——敏感操作需经审批链 |

### 5.1.1 7 维度决策顺序(纵深防御)

```
访问请求 →
  ① TenantScope 通过?  → No: 拒绝
  → Yes ↓
  ② RBAC 角色匹配?    → No: 拒绝
  → Yes ↓
  ③ MatterScope 通过?  → No: 拒绝
  → Yes ↓
  ④ DataLevel 满足?    → No: 拒绝
  → Yes ↓
  ⑤ StageGate 阶段允许? → No: 拒绝 / 限流
  → Yes ↓
  ⑥ ABAC 属性检查?     → No: 拒绝 / 二次认证
  → Yes ↓
  ⑦ ApprovalFlow 需审批? → Yes: 进入审批,完成后执行
  → No: 执行
```

> **设计原则**:**纵深防御** —— 7 个维度任一不通过即拒绝,**且每一步审计日志全留痕**。

## 5.2 数据密级(6 级)

| 级别 | 代码 | 含义 | 典型内容 |
|------|------|------|----------|
| **L1** | `L1_PUBLIC` | 公开内容 | 律所简介、律师介绍(已脱敏)、服务产品目录、公开案例(已脱敏) |
| **L2** | `L2_FIRM_INTERNAL` | 律所内部 | 律所规章、内部通知、培训资料、案件统计、营收数据(脱敏) |
| **L3** | `L3_MATTER_TEAM` | 案件团队 | 案件事实、证据、策略、律师内部备注、案件时间线 |
| **L4** | `L4_CLIENT_CONFIDENTIAL` | 客户秘密 | 客户商业秘密、合同底价、客户底线、客户决策信息 |
| **L5** | `L5_HIGHLY_CONFIDENTIAL` | 高敏案件 | 涉外、刑事、上市公司、重大民商事、群体性案件、个人隐私敏感 |
| **L6** | `L6_AI_RESTRICTED` | 限制 AI 处理 | **任何不得进入 AI 模型的数据**(L4 / L5 子集或独立标注) |

### 5.2.1 L6 限制 AI 处理 的设计要点(关键,工程硬约束)

> L6 是 **合规与 AI 工程的关键边界** —— 即使 L4 / L5 数据,只要标注 L6,**律时数字人、文档智能、AI 研究助手均不得读取、不得摘要、不得向量化、不得作为 prompt 输入**。

| 约束项 | 实现方式 |
|--------|----------|
| 律时对话 | 消息前置过滤,命中 L6 关键词 / 标签时直接拒绝 + 提示转人工 |
| 文档上传 | L6 文档 **不入向量库**;不入 AI 训练数据;不入任何模型上下文 |
| AI 调用审计 | 所有 AI 调用记录 `data_levels` 字段;含 L6 即告警 |
| 数据导出 | L6 数据 **不可导出至 AI 服务端**(本地推理或脱敏后方可) |

### 5.2.2 数据密级 × 角色 访问矩阵(框架,待 4.6 详细化)

| 数据密级 \ 角色 | 平台 | 合伙 | 主办 | 协办 | 助理 | 风控 | 运营 | 客户 | 支持企业 |
|----------------|------|------|------|------|------|------|------|------|----------|
| L1 公开 | R | R | R | R | R | R | R | R | R |
| L2 律所内部 | R | R | R | R | R | R | R(8 项) | — | R(授权) |
| L3 案件团队 | — | R | R | R | R | R(过程) | — | — | R(授权) |
| L4 客户秘密 | — | R | R | R | R(部分) | R(过程) | — | R(本人) | — |
| L5 高敏案件 | — | R | R(逐案) | R(逐案) | R(逐案) | R(过程) | — | R(本人) | — |
| L6 AI 限制 | — | R(可读) | R(可读) | R(可读) | R(可读) | R(可读) | — | R(本人) | — |

> **图例**:R = 可读, — = 不可读
> L6 列特指 **"人可读,但 AI 不可处理"**
> "R(过程)" 指仅在风控审查节点可见
> "R(授权)" 指需律所显式授权
> "R(8 项)" 指 5.3 节运营可见清单
> "R(本人)" 指客户仅可看本人所属案件

## 5.3 运营人员权限边界

### 可见(8 项)

| 序号 | 资源 | 用途 | 数据密级 |
|------|------|------|----------|
| 1 | 线索基础信息 | 线索分诊、跟进 | L1 / L2 |
| 2 | 客户预约信息 | 排期与到访 | L2 |
| 3 | 来源渠道 | 渠道分析与归因 | L2 |
| 4 | 客户反馈 | 服务改进 | L2 |
| 5 | 服务产品信息 | 报价与匹配 | L1 |
| 6 | 报价卡状态 | 进度跟进 | L2 |
| 7 | 客户满意度 | 客户成功 | L2 |
| 8 | 工单状态 | 售后响应 | L2 |

### 不可见(8 项)

| 序号 | 资源 | 数据密级 | 不可见原因 |
|------|------|----------|------------|
| 1 | 案件策略 | L3 | 律师工作内容 |
| 2 | 法律意见 | L3 / L4 | 律师交付物 |
| 3 | 证据分析 | L3 / L5 | 律师工作内容 |
| 4 | 客户底线 | L4 | 客户秘密 |
| 5 | 律师内部备注 | L3 | 律师工作内容 |
| 6 | 文书草稿 | L3 | 律师工作内容 |
| 7 | 客户商业秘密 | L4 | 客户秘密 |
| 8 | 高敏案件材料 | L5 | 案件敏感性 |

> **设计原则**:
> - 运营人员 **"重流程、轻内容"** —— 可推动流程,但不接触案件实质
> - 直接对应 [2.1 第 2 条](./02-规则商业技术三重定位.md) **"运营人员不得作法律判断"**
> - 系统层面:UI 默认隐藏;API 层接口 403;审计层全留痕

## 5.4 客户侧权限边界

### 可见(8 项)

| 序号 | 资源 | 用途 | 数据密级 |
|------|------|------|----------|
| 1 | 案件阶段 | 进度透明化 | L3(已脱敏) |
| 2 | 客户待办 | 客户配合事项 | L4(本人) |
| 3 | 已审核进度报告 | 阶段性交付 | L3 → L1 转换后 |
| 4 | 已交付文件 | 律师交付物 | L3 → L1 转换后 |
| 5 | 会议日程 | 协同安排 | L3(本人) |
| 6 | 费用节点 | 账单透明 | L2(本人) |
| 7 | 服务产品说明 | 服务边界 | L1 |
| 8 | 已确认风险揭示 | 合规留痕 | L4(本人) |

### 不可见(7 项)

| 序号 | 资源 | 数据密级 | 不可见原因 |
|------|------|----------|------------|
| 1 | 内部策略 | L3 | 律师工作内容 |
| 2 | 律师审核意见 | L3 / L4 | 律师交付物 |
| 3 | AI 内部分析 | L3 | 律时工作内容(非交付物) |
| 4 | 对客户不利的风险研判 | L3 / L4 | 律师执业策略 |
| 5 | 文书修改过程 | L3 | 律师工作过程 |
| 6 | 其他客户资料 | L2 / L3 | 客户隔离 |
| 7 | 律师工时内部备注 | L3 | 律师工作内容 |

> **设计原则**:
> - 客户 **"看交付物,不看工作过程"** —— 已审核的成果可看,中间过程不可见
> - 保护了 **律师执业策略独立性** 和 **客户之间的隔离**
> - 客户看到的"已审核进度报告" 是 **律师主动选择披露** 的内容,系统不强制全披露

## 5.5 7 维度在典型场景中的协同

### 场景 1:运营人员查询某案件工单 → ✅ 允许

```
TenantScope  : 同一 Firm ✓
RBAC         : OPERATION_USER ✓
MatterScope  : 该案件 ID 命中运营白名单 ✓
DataLevel    : 工单状态 = L2 ✓ (在 5.3 可见清单)
StageGate    : 当前阶段允许运营介入 ✓
ABAC         : 同一工作时间、同一设备 ✓
ApprovalFlow : 常规动作,无需审批 ✓
→ 允许
```

### 场景 2:客户下载某文书草稿 → ❌ 拒绝(数据密级不符)

```
TenantScope  : 同一 Firm ✓
RBAC         : CLIENT_OWNER ✓
MatterScope  : 客户所属案件 ✓
DataLevel    : 文书草稿 = L3 ✗
→ 拒绝(数据密级不符)
```

### 场景 3:律师助理访问 L5 高敏案件 → 🔒 进入审批流

```
TenantScope  : 同一 Firm ✓
RBAC         : LEGAL_ASSISTANT ✓
MatterScope  : 命中具体案件 ✓
DataLevel    : L5 ✗(助理默认无 L5)
ApprovalFlow : 需主办律师 / 合伙人审批 ✗
→ 进入审批流
```

### 场景 4:律时读取 L6 客户秘密文档 → ❌ 拒绝(L6 AI 限制)

```
TenantScope  : 同一 Firm ✓
RBAC         : AI 助理 ✓
MatterScope  : 命中具体案件 ✓
DataLevel    : L6 ✗(AI 限制)
→ 拒绝(无论哪个 AI 角色,L6 一律禁读)
```

### 场景 5:支持企业运营人员查看客户商业秘密 → ❌ 拒绝(数据范围未授权)

```
TenantScope  : 同一 Firm ✓
RBAC         : SUPPORT_OPERATOR ✓
MatterScope  : 律所未授权该支持企业访问此案件 ✗
→ 拒绝(数据范围未授权)
```

### 场景 6:律师在非工作时间下载 L5 高敏案件 → 🔐 二次认证

```
TenantScope  : 同一 Firm ✓
RBAC         : LEAD_LAWYER ✓
MatterScope  : 命中具体案件 ✓
DataLevel    : L5 ✓
StageGate    : ✓
ABAC         : 非常规工作时间 ✗
→ 触发二次认证(短信 / 邮件 / 二次确认)
```
