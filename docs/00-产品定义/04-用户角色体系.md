# 第 4 章 · 用户角色体系

> 本章定义 LM Unity · Counsel + 律时 系统的全部 **5 类 33 个角色**。角色是后续权限(RBAC)、数据隔离(参见 3.2 节)、合规审计(参见 2.1 节)的最小单位。
>
> 任何"**角色 × 数据 × 操作**"三元组,在 4 级数据隔离与 10 条合规底线的约束下,必须 **可审计、可追溯、可回放**。

---

## 4.0 角色总览

| 类别 | 数量 | 数据归属 | 法律判断权限 |
|------|------|----------|--------------|
| 4.1 平台级 | 4 | Tenant | 无 |
| 4.2 律所组织级 | 10 | Firm | 受限(仅合伙人有强权限) |
| 4.3 律师业务级 | 8 | Firm / Workspace / Tenant | **强**(交付主体) |
| 4.4 客户侧 | 7 | Firm(只读本人) | 无 |
| 4.5 法律服务支持型企业 | 4 | Firm(支持范围) | **无**(强约束) |
| **合计** | **33** | —— | —— |

> **核心红线**(来自 [2.1 第 2 条](./02-规则商业技术三重定位.md)):
> **律师 / 律所** 是法律服务交付主体;**运营 / 客服 / 支持型企业人员** 不得作法律判断。本章所有角色都按此原则标注"法律判断权限"。

## 4.1 平台级角色

| 角色代码 | 角色名称 | 数据归属 | 法律判断权限 |
|----------|----------|----------|--------------|
| `PLATFORM_SUPER_ADMIN` | 平台超级管理员 | Tenant | 无 |
| `PLATFORM_OPERATOR` | 平台运营人员 | Tenant | 无 |
| `PLATFORM_COMPLIANCE` | 平台合规审核人员 | Tenant | 无 |
| `PLATFORM_TECH_SUPPORT` | 技术支持 | Tenant | 无 |

> **设计说明**:
> - 平台级角色是 **SaaS 平台的运营 / 运维 / 合规管理方**
> - **不进入任何具体律所的业务数据**(受 4 级数据隔离保护)
> - 所有平台操作均留痕,接受第三方审计

| `OPERATION_USER` | 运营人员 | Firm | 无 |

> **设计说明**:

> - 其余 8 个角色 **一律无法律判断权限** —— 这是 2.1 第 2 条强约束的体现

## 4.3 律师业务角色

| 角色代码 | 角色名称 | 数据归属 | 法律判断权限 |
|----------|----------|----------|--------------|
| `RESPONSIBLE_PARTNER` | 负责合伙人 | Firm / Workspace | **强** |
| `LEAD_LAWYER` | 主办律师 | Firm / Workspace | **强** |
| `ASSOCIATE_LAWYER` | 协办律师 | Firm / Workspace | 中(协同办案) |
| `LEGAL_ASSISTANT` | 律师助理 | Firm / Workspace | 无 |
| `INTERN` | 实习生 | Firm / Workspace | 无 |
| `SOLO_LAWYER` | 独立律师 | Tenant(个人) | **强** |
| `EXTERNAL_LAWYER` | 外部协作律师 | Firm(协作范围) | 中(限于协作) |
| `EXPERT_ADVISOR` | 外部专家 | Firm(咨询范围) | 中(限于咨询) |

> **设计说明**:
> - **交付主体角色**:`RESPONSIBLE_PARTNER` / `LEAD_LAWYER` / `SOLO_LAWYER`
> - `SOLO_LAWYER` 是 **个人律师版** 的核心角色,数据归属直接到 Tenant(无 Firm 概念)
> - `EXTERNAL_LAWYER` 与 `EXPERT_ADVISOR` 的"法律判断权限"为中,需配合"协作 / 咨询"场景使用,系统应自动记录其意见作为 **参考而非最终结论**

| 角色代码 | 角色名称 | 数据归属 | 法律判断权限 |
|----------|----------|----------|--------------|
| `CLIENT_OWNER` | 客户主联系人 | Firm(只读本人) | 无 |
| `CLIENT_CONTACT` | 客户联系人 | Firm(只读本人) | 无 |
| `CLIENT_LEGAL` | 客户法务 | Firm(只读本人) | 无 |
| `CLIENT_FINANCE` | 客户财务 | Firm(只读本人) | 无 |
| `CLIENT_DECISION_MAKER` | 客户决策人 | Firm(只读本人) | 无 |
| `INDIVIDUAL_CLIENT` | 个人客户 | Firm(只读本人) | 无 |

> **设计说明**:
> - 客户侧角色 **一律无法律判断权限**
> - 数据访问范围:**只读本人所属案件的进度、文档、账单**(由律所 RBAC 配置)
> - `INDIVIDUAL_CLIENT` 是 C 端个人客户,主要使用"客户门户 + 律时客户助理"

## 4.5 法律服务支持型企业角色

| 角色代码 | 角色名称 | 数据归属 | 法律判断权限 |
|----------|----------|----------|--------------|
| `SUPPORT_ORG_ADMIN` | 支持企业管理员 | Firm(支持范围) | 无 |
| `SUPPORT_OPERATOR` | 支持企业运营人员 | Firm(支持范围) | 无 |
| `SUPPORT_QA` | 支持企业质检人员 | Firm(支持范围) | 无 |
| `SUPPORT_DATA_ANALYST` | 支持企业数据分析人员 | Firm(支持范围) | 无 |

### 4.5.1 强约束(来自 2.1 第 2 条 + 本节)

> 该类角色可承担 **运营、CRM、预约、回访、满意度追踪、质检抽检、培训和知识库管理** 等支持功能,但必须严格禁止进入 **法律判断、案件收费和结果分成** 领域;这一点也符合附件文章所提出的"**法律服务支持型企业**"应被制度化嵌入正当供给体系、但不得侵入律师职业边界的治理思路。

### 4.5.2 系统级硬约束实现

| 约束项 | 系统实现 |
|--------|----------|
| 不得作法律判断 | 角色无"法律意见 / 文书终审 / 案件定性"操作权限;UI 上隐藏入口 |
| 不得参与案件收费 | 财务模块无该角色写权限;账单流程强校验 |
| 不得参与结果分成 | 财务模块禁止"按结果 / 标的分成"科目;审计月报自动告警 |
| 数据访问范围 | 律所授权(显式勾选"允许该支持企业访问以下案件 / 字段") |
| 操作留痕 | 所有动作入审计日志,律所可随时审计与回溯 |

## 4.6 角色 × 资源 操作矩阵(框架,待详细填充)

> 此表为后续 **RBAC 权限矩阵** 的输入框架,完整版需按业务逐项展开。

| 角色 \ 资源 | 案件 | 客户 | 文档 | 财务 | 律时 AI | 知识库 |
|------------|------|------|------|------|---------|--------|
| `MANAGING_PARTNER` | CRUD | CRUD | CRUD | R | C(指令) | CRUD |
| `RISK_PARTNER` | R + 标注 | R | R | R | C(风控指令) | R |
| `LEAD_LAWYER` | CRUD(本人) | CRUD(本人) | CRUD(本人) | R | C(指令) | CRUD(本人) |
| `ASSOCIATE_LAWYER` | RU(协作) | R(协作) | RU(协作) | R | C(指令) | R |
| `LEGAL_ASSISTANT` | RU(部分) | RU(部分) | RU(部分) | R | C(白名单) | R |
| `INTERN` | R(部分) | R(部分) | R(部分) | — | C(白名单) | R |
| `SOLO_LAWYER` | CRUD(本人) | CRUD(本人) | CRUD(本人) | R | C(指令) | CRUD(本人) |
| `EXTERNAL_LAWYER` | RU(协作) | R(协作) | RU(协作) | — | C(白名单) | — |
| `EXPERT_ADVISOR` | R(咨询) | — | R(咨询) | — | C(白名单) | — |
| `CLIENT_OWNER` | R(本人) | R(本人) | R(本人) | R(本人) | C(咨询) | — |
| `INDIVIDUAL_CLIENT` | R(本人) | R(本人) | R(本人) | R(本人) | C(咨询) | — |
| `SUPPORT_OPERATOR` | — | R(授权) | — | — | C(白名单) | — |
| `SUPPORT_QA` | R(抽检) | R(抽检) | R(抽检) | — | — | — |
| `SUPPORT_DATA_ANALYST` | R(脱敏) | R(脱敏) | R(脱敏) | R(脱敏) | — | R(脱敏) |
| `PLATFORM_*` | — | — | — | — | C(运维) | — |

> **图例**:C = Create, R = Read, U = Update, D = Delete
> "本人 / 协作 / 授权 / 抽检 / 脱敏" 等限定词表示 **数据范围**,由 RBAC + 4 级数据隔离联合控制
