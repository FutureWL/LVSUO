# 第 8 章 · 数据库设计

> 本章给出 15 张 **核心表** 的 DDL。开发团队可在详细设计阶段拆分枚举、索引、外键和审计表。
>
> **命名约定**:
> - 表名:`snake_case`,复数
> - 主键:`id BIGINT`
> - 租户列:每张业务表必有 `tenant_id`
> - 时间戳:必有 `created_at` / `updated_at`(审计要求)
> - 状态字段:使用 `VARCHAR`,枚举值见各表注释

---

## 8.1 租户表 `tenants`

```sql
CREATE TABLE tenants (
    id              BIGINT PRIMARY KEY,
    tenant_name     VARCHAR(255) NOT NULL,
    tenant_type     VARCHAR(50),  -- FIRM / SOLO / TEAM / SUPPORT_ORG
    deployment_mode VARCHAR(50),  -- SAAS / PRIVATE / PERSONAL
    status          VARCHAR(50),
    created_at      DATETIME,
    updated_at      DATETIME
);
```

> **后续设计要点**(待详细设计阶段补充)
>
> - **索引设计**:所有 `tenant_id` 必须建索引;高频查询字段 `(tenant_id, status, created_at)` 复合索引
> - **外键约束**:业务表之间的外键建议 **逻辑外键**(应用层维护),避免硬约束影响 SaaS 性能
> - **审计表**:每张表对应 `_audit` 后缀的审计表,记录 `INSERT / UPDATE / DELETE` 前镜像
> - **多租户隔离**:`tenant_id` 必为查询条件,建议用 PostgreSQL Row-Level Security 强制
> - **L6 数据**:`confidentiality_level = 'L6_AI_RESTRICTED'` 的记录不进向量库 / AI 训练数据
> - **数据生命周期**:GDPR / PIPL 要求的"被遗忘权"实现(详见 13.2 审计日志)
